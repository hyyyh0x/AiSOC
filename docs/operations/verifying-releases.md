# Verifying AiSOC Releases

You run AiSOC images next to your crown jewels, so you should verify what you download. This page documents how to check release integrity today and what stronger attestations are landing.

## Available today

### Source tarball checksums (`release.yml`)

Every tagged release publishes the source tarball plus SHA-256 and SHA-512 checksums as release assets.

```bash
# Download the tarball and checksums from the release, then:
sha256sum -c aisoc-<version>.tar.gz.sha256
sha512sum -c aisoc-<version>.tar.gz.sha512
```

### Source SBOM (`release.yml`)

Each release attaches an SPDX SBOM for the source tree (generated by `anchore/sbom-action`). Import it into your SCA tooling to inventory dependencies.

### Browser-extension signatures (`build-extensions.yml`)

The osquery browser-extension binaries are signed with `cosign` (keyless / OIDC) on `ext-v*` tags. Verify with:

```bash
cosign verify-blob \
  --certificate <artifact>.pem \
  --signature <artifact>.sig \
  --certificate-identity-regexp 'https://github.com/beenuar/AiSOC/.*' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  <artifact>
```

### Container image signatures (`publish-images.yml`, `release.yml`)

Every `ghcr.io/beenuar/*` service image pushed to `main` or a release tag is signed with `cosign` (keyless / OIDC). Verify before running:

```bash
cosign verify \
  --certificate-identity-regexp 'https://github.com/beenuar/AiSOC/.*' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  ghcr.io/beenuar/aisoc-core-api:<version>
```

### Per-image CycloneDX SBOM attestation

Each image carries a CycloneDX SBOM attached as a cosign attestation (generated by `syft` at push time):

```bash
cosign verify-attestation \
  --type cyclonedx \
  --certificate-identity-regexp 'https://github.com/beenuar/AiSOC/.*' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  ghcr.io/beenuar/aisoc-core-api:<version> \
  | jq -r '.payload' | base64 -d | jq '.predicate'
```

### SLSA build provenance + SPDX SBOM (BuildKit attestations)

Images are built with `provenance: mode=max` and `sbom: true`, so BuildKit attaches SLSA provenance and an SPDX SBOM directly to the image manifest:

```bash
docker buildx imagetools inspect ghcr.io/beenuar/aisoc-core-api:<version> \
  --format '{{ json .Provenance }}'
docker buildx imagetools inspect ghcr.io/beenuar/aisoc-core-api:<version> \
  --format '{{ json .SBOM }}'
```

### SHA-pinned GitHub Actions

Every third-party action in `.github/workflows/` is pinned to a full commit SHA (the tag is retained as a trailing comment). A tag-hijack of an upstream action cannot change what our CI executes.

The enforcing gates are tracked in [`docs/audit/CLAIM_TO_GATE_MATRIX.md`](../audit/CLAIM_TO_GATE_MATRIX.md) and `ROADMAP.md`.
